## tmuxのレイヤー構造を理解する

tmuxには最初は混乱しやすい4つのレイヤーがあります:

```
┌─────────────────────────────────────────┐
│ クライアント(ローカルターミナル/PSMUX)    │  ← 複数持てる
│  ↓ 接続先                               │
│ セッション(永続的、サーバーで実行)        │  ← 1つのセッションを共有
│  ↓ 含む                                 │
│ ウィンドウ(タブのような)                 │  ← これを積極的に使う
│  ↓ 含む                                 │
│ ペイン(ウィンドウ内の分割セクション)      │  ← 控えめに使う
└─────────────────────────────────────────┘
```

**重要な洞察**: 複数のクライアントが*同じ*セッションに接続できます。これは次を意味します:
- 新しいターミナルウィンドウを開いて接続 → 同じセッション、同じウィンドウ
- VS Codeターミナル接続 → 同じセッションを使用可能
- すべてのクライアントが同じウィンドウとペインを見る
- 1つのクライアントが切断しても、セッションは他のクライアントのために存続

**推奨アプローチ**: **1つまたは2つのセッション**に**複数のウィンドウ**を使用し、複数のセッションは使用しない。

## 推奨セットアップ

### 接続エイリアス/スクリプト

毎回`ssh ... ; tmux attach`と入力する代わりに、メインセッションに自動的にアタッチする簡単な接続を作成します:

**bash/zshの場合**(ローカルマシンの`~/.bashrc`または`~/.zshrc`に追加):

```bash
# AWSに接続し'main'セッションにアタッチ(存在しない場合は作成)
alias aws-dev='ssh -i ~/.ssh/your-aws-key.pem <ユーザー名>@<EC2ホスト名> -t "tmux new-session -A -s main"'
```

**またはスクリプトを作成**(`~/bin/aws-dev`):

```bash
#!/bin/bash
# AWS EC2に接続しメインtmuxセッションにアタッチ
ssh -i ~/.ssh/your-aws-key.pem <ユーザー名>@<EC2ホスト名> -t "tmux new-session -A -s main"
```

実行可能にする:
```bash
chmod +x ~/bin/aws-dev
```

**これが行うこと**:
- `tmux new-session -A -s main` → "main"という名前のセッションが存在すればアタッチ、なければ作成
- この方法で接続するすべてのターミナルウィンドウ/タブが同じセッションを共有
- 切断して再接続すると、まったく同じ状態に戻る

## 基本的なワークフロー

### 1日の始め方

1. **最初のターミナルから接続**:
   ```bash
   aws-dev  # (上記のエイリアスを使用)
   ```
   これで"main"セッションを作成またはアタッチ

2. **異なるタスク用のウィンドウを作成**:
   ```
   Ctrl+^ c    # ML学習スクリプト用のウィンドウを作成
   Ctrl+^ c    # モニタリング用のウィンドウを作成(htop/nvidia-smi)
   Ctrl+^ c    # git/編集用のウィンドウを作成
   ```

3. **必要に応じてウィンドウを切り替え**:
   ```
   Ctrl+^ 0    # ウィンドウ0へ移動
   Ctrl+^ 1    # ウィンドウ1へ移動
   Ctrl+^ n    # 次のウィンドウ
   Ctrl+^ p    # 前のウィンドウ
   ```

### 追加クライアントを開く

**2つ目のターミナルウィンドウ**を開いて再度`aws-dev`を実行すると:
- ✅ **同じセッション**に接続
- ✅ **同じウィンドウ**(0, 1, 2など)が表示される
- ✅ 両方のターミナルが同期状態を保つ
- ✅ 1つが切断しても、もう1つは継続

**これが別々のセッションより優れている理由**:
- どのセッションにどの作業があるか混乱しない
- すべての作業が1か所にある
- 異なるターミナルウィンドウ間で状態を簡単に共有
- ネットワーク問題の後に再接続しても、すべてが残した場所にある

## ウィンドウ管理コマンド

注意: プレフィックスは**Ctrl+^**(Ctrlを押しながら^を押す)

### ウィンドウの作成と閉じる

| コマンド | 説明 |
| --------- | ------------- |
| `Ctrl+^ c` | 新しいウィンドウを**作成** |
| `Ctrl+^ ,` | 現在のウィンドウを**名前変更** |
| `Ctrl+^ &` | 現在のウィンドウを**終了**(確認を求める) |
| `exit` | 現在のペイン/ウィンドウを閉じる(最後のペインの場合) |

### ウィンドウのナビゲーション

| コマンド | 説明 |
| --------- | ------------- |
| `Ctrl+^ 0-9` | ウィンドウ**番号**へ移動(0-9) |
| `Ctrl+^ n` | **次の**ウィンドウへ移動 |
| `Ctrl+^ p` | **前の**ウィンドウへ移動 |
| `Ctrl+^ l` | **最後に**使用したウィンドウへ移動 |
| `Ctrl+^ w` | すべてのウィンドウを**リスト**(矢印で選択) |
| `Ctrl+^ f` | 名前でウィンドウを**検索** |

### ウィンドウリスト

ターミナルの下部に次のようなものが表示されます:
```
[main] 0:bash  1:training*  2:monitor-  3:code
```

- `[main]` = セッション名
- `0:bash` = ウィンドウ番号と名前
- `*` = 現在のウィンドウ
- `-` = 最後にいたウィンドウ

### ウィンドウで作業を整理

**ML学習のセットアップ例**:
```
ウィンドウ 0 (bash)       → 一般的なシェル、ファイル管理
ウィンドウ 1 (train)      → 学習スクリプトの実行
ウィンドウ 2 (monitor)    → htop / nvidia-smi / watchコマンド
ウィンドウ 3 (jupyter)    → Jupyterノートブックサーバー
ウィンドウ 4 (code)       → コード編集 / git操作
```

目的を覚えるためにウィンドウの名前を変更:
```bash
Ctrl+^ ,              # 次に新しい名前を入力、例: "training"
```

## 複数セッションを使用する場合

**一般的には**多くのセッションの作成を避けてください。しかし、有効なユースケースがあります:

### 2セッション構成(推奨最大)

```bash
# セッション1: "main" - 手動ターミナル作業用
tmux new-session -A -s main

# セッション2: "vscode" - VS Codeターミナル専用
tmux new-session -A -s vscode
```

**なぜ別々のセッション?**
- VS Codeターミナルの自動アタッチが手動作業を妨げない
- 手動ターミナルセッションに影響を与えずにVS Codeを閉じられる
- 各セッションが独立したウィンドウ番号を持つ

**セッション間を切り替える**(アタッチ時):
```
Ctrl+^ s    # セッションリストを表示、矢印で選択
Ctrl+^ (    # 前のセッションへ切り替え
Ctrl+^ )    # 次のセッションへ切り替え
```

### 新しいセッションを作成すべきでない場合

❌ **作成しない**:
- 異なるプロジェクト → 代わりにウィンドウを使用
- 異なるタスク → 代わりにウィンドウを使用
- 各ターミナルウィンドウ → 同じセッションに接続

✅ **ウィンドウを作成する**:
- 異なるプロジェクト
- 異なるタスク(学習、モニタリング、編集)
- 個別の長時間実行プロセス

## よくあるシナリオ

### シナリオ1: 長時間学習ジョブ

```bash
# メインセッションに接続
aws-dev

# 学習専用ウィンドウを作成
Ctrl+^ c
Ctrl+^ ,
# 入力: "training"

# 学習を開始
python train_model.py

# モニタリングウィンドウを作成
Ctrl+^ c
Ctrl+^ ,
# 入力: "monitor"

# GPU使用率を監視
watch -n 1 nvidia-smi

# ログを確認するために学習ウィンドウに戻る
Ctrl+^ p  # またはCtrl+^の後にウィンドウ番号
```

**安全に切断**: ターミナルを閉じるか`Ctrl+^ d`を押す
- 学習は実行し続ける
- `aws-dev`でいつでも再接続
- すべてが残した通りの状態

### シナリオ2: 複数のターミナルウィンドウ

学習実行中に何かをチェックしたい:

```bash
# 元のターミナル: ウィンドウ1で学習が実行中

# ラップトップで新しいターミナルウィンドウを開く
aws-dev  # 同じ"main"セッションに接続

# 同じウィンドウが表示される!
# クイックタスク用に新しいウィンドウを作成
Ctrl+^ c

# 作業を行う...

# 完了したらこのターミナルを閉じる
# ウィンドウ1の学習は影響を受けずに継続
```

### シナリオ3: VS Code + 手動ターミナル

```bash
# 手動ターミナル
ssh <host> -t "tmux new-session -A -s main"

# VS Code Remote-SSHターミナル(settings.jsonで設定)
"terminal.integrated.profiles.linux": {
  "tmux-vscode": {
    "path": "tmux",
    "args": ["new-session", "-A", "-s", "vscode"]
  }
}
```

これで以下が得られます:
- `main`セッション: SSHターミナル作業用
- `vscode`セッション: VS Code統合ターミナル用
- VS Codeを閉じても競合なし

### シナリオ4: ネットワーク中断

```bash
# ウィンドウ1で作業中(学習)
python long_running_job.py

# ネットワークが切断!
# (セッションはサーバーで継続)

# 後で再接続
aws-dev

# 戻った! 同じウィンドウ、学習はまだ実行中
Ctrl+^ 1  # 学習ウィンドウへ移動
# ログをチェック、まだ実行していることを確認
```

## クイックリファレンスカード

### 必須コマンド(使用の90%)

```bash
# 接続
tmux new-session -A -s main     # "main"にアタッチ(または作成)

# ウィンドウ
Ctrl+^ c        新しいウィンドウを作成
Ctrl+^ ,        ウィンドウの名前を変更
Ctrl+^ 0-9      ウィンドウ番号へ移動
Ctrl+^ n        次のウィンドウ
Ctrl+^ p        前のウィンドウ
Ctrl+^ w        ウィンドウをリスト

# セッション
Ctrl+^ d        デタッチ(切断するが実行し続ける)
Ctrl+^ s        セッションをリスト(複数使用時)

# ヘルプ
Ctrl+^ ?        すべてのキーバインディングを表示
```

### 実行中の確認

```bash
# tmuxの外から
tmux ls                          # セッションをリスト

# tmuxの中から
Ctrl+^ w                         # 現在のセッションのウィンドウをリスト
Ctrl+^ s                         # すべてのセッションをリスト
```

## 成功のためのヒント

1. **シンプルに始める**: 1つのセッション(`main`)と複数のウィンドウを使用
2. **ウィンドウに名前を付ける**: `Ctrl+^ ,`で何がどこにあるか覚えやすい
3. **ターミナルタブを使わない**: tmuxウィンドウを"タブ"にする
4. **セッションを信頼する**: 切断してもすべてが実行し続ける
5. **後でカスタマイズ**: 慣れたら、ペイン、カスタムレイアウトなどを探索
